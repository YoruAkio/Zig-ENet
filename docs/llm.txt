1. Identity

Project name:
Zig ENet

Main purpose:
A Zig-first ENet-style UDP networking library focused on Growtopia private server development.

Language:
Zig

Design direction:
- keep the public API Zig-native
- preserve ENet-style protocol behavior where it matters
- support Growtopia-specific wire behavior through explicit new-packet flags
- favor testable protocol logic over platform-coupled design


2. What This Project Is

This project is a Zig implementation of an ENet-style networking stack.
It is not a raw C binding layer.
It exposes Zig types such as Host, Peer, Packet, Address, and Event.

The project supports:
- reliable packets
- unreliable packets
- unsequenced packets
- fragmentation and reassembly
- acknowledgements and resend timeouts
- RTT tracking
- bandwidth and throttle control paths
- optional checksum support
- optional compression support
- Unix and Windows UDP backends
- Growtopia-oriented new-packet behavior


3. Core Mental Model

Think about the library in layers:

- config layer
  host settings, new-packet flags, checksum callback, compression callbacks

- wire layer
  header layout, command encoding, command sizes, checksum field placement

- protocol layer
  sequencing, fragmentation, throttle math, connect payload parsing

- host layer
  service loop, send queue, receive path, event generation, resend logic

- transport layer
  UDP socket backend or mock transport

If you need to explain or change behavior, always identify which layer owns it first.

6. Most Important Public Types

Address
- represents a network address
- used for bind addresses and remote peer addresses

HostConfig
- config passed into Host.init or Host.withTransport
- important fields:
  - address
  - peer_limit
  - channel_limit
  - bandwidth
  - mtu
  - maximum_packet_size
  - maximum_waiting_data
  - using_new_packet
  - using_new_packet_for_server
  - checksum_fn
  - compression

Host
- main owner of peers, queues, events, and service loop
- main methods:
  - init
  - withTransport
  - deinit
  - connect
  - broadcast
  - flush
  - service

Peer
- represents a remote peer known by a host
- main methods:
  - configureThrottle
  - setTimeout
  - send
  - disconnect

Packet
- heap-allocated payload container
- uses explicit retain and release semantics

Event
- returned by Host.service
- event.type is one of:
  - none
  - connect
  - disconnect
  - receive


7. Runtime Flow

Normal lifecycle:

1. create a transport
2. create a Host
3. connect to a remote address or wait for incoming traffic
4. call Host.service in a loop
5. react to Event values
6. send Packet values through Peer.send
7. optionally call Host.flush for immediate transmission
8. release received packets after handling them

Key rule:
Host.service both receives incoming packets and advances internal protocol state.


8. Ownership Rules

These rules are important. Do not describe them loosely.

Packet ownership:
- Packet is reference-counted with retain and release
- queued outgoing commands retain packet references
- sent reliable tracking retains packet references until command lifecycle completes
- received Event.packet values should be released by the caller

Host ownership:
- Host owns peer storage
- Host.deinit cleans up peers, queued commands, tracked reliable commands, queued events, transport, and compression context

Peer ownership:
- Peer is owned by Host
- do not manually free Peer

Transport ownership:
- if attached to Host, transport is deinitialized by Host.deinit


9. Important Invariants

These are useful when answering questions or changing code:

- HostConfig.channel_limit is clamped through protocol helpers
- peer ids must stay within constants.protocol_maximum_peer_id
- packet size must not exceed HostConfig.maximum_packet_size
- checksum behavior depends on HostConfig.checksum_fn
- compression behavior depends on HostConfig.compression callbacks
- Growtopia integrity validation only matters when using_new_packet_for_server is enabled
- Host.flush sends acknowledgements, resends, then queued commands
- Host.service may return one immediate event and may also enqueue later events internally
- wire size questions should use src/wire.zig helpers, not raw @sizeOf assumptions
- Packet.release may destroy the packet immediately when reference_count reaches zero


10. New Packet Flag Notes

using_new_packet
- enables outgoing new-packet header behavior
- affects outgoing header encoding and fragment sizing
- use when this host should send the newer packet format

using_new_packet_for_server
- enables incoming server-side new-packet parsing
- enables Growtopia integrity validation on receive
- use when this host should accept the server-side Growtopia packet layout

12. Example: Basic Server

This is the simplest mental model for a server:

```zig
const std = @import("std");
const builtin = @import("builtin");
const zigenet = @import("zigenet");

pub fn main() !void {
    const bind_address = zigenet.Address.any(17091);
    const transport = if (builtin.os.tag == .windows)
        try zigenet.transport.windows.UdpSocket.bind(std.heap.page_allocator, bind_address)
    else
        try zigenet.transport.unix.UdpSocket.bind(std.heap.page_allocator, bind_address);

    var host = try zigenet.Host.withTransport(std.heap.page_allocator, .{
        .address = bind_address,
        .peer_limit = 32,
        .channel_limit = 2,
        .using_new_packet = false,
        .using_new_packet_for_server = false,
    }, transport);
    defer host.deinit();

    while (true) {
        if (try host.service(16)) |event| {
            switch (event.type) {
                .connect => {},
                .disconnect => {},
                .receive => {
                    defer event.packet.?.release();
                },
                else => {},
            }
        }
    }
}
```

13. Example: Basic Client

```zig
const std = @import("std");
const builtin = @import("builtin");
const zigenet = @import("zigenet");

pub fn main() !void {
    const transport = if (builtin.os.tag == .windows)
        try zigenet.transport.windows.UdpSocket.bind(std.heap.page_allocator, null)
    else
        try zigenet.transport.unix.UdpSocket.bind(std.heap.page_allocator, null);

    var host = try zigenet.Host.withTransport(std.heap.page_allocator, .{
        .peer_limit = 1,
        .channel_limit = 2,
        .using_new_packet = false,
        .using_new_packet_for_server = false,
    }, transport);
    defer host.deinit();

    const peer = try host.connect(
        zigenet.Address.fromIpv4Octets(.{ 127, 0, 0, 1 }, 17091),
        2,
        0,
    );

    try host.flush();

    while (true) {
        if (try host.service(16)) |event| {
            switch (event.type) {
                .connect => {
                    var packet = try zigenet.Packet.create(
                        std.heap.page_allocator,
                        "hello from zig",
                        14,
                        zigenet.constants.packet_flag_reliable,
                    );
                    defer packet.release();
                    packet.retain();
                    try event.peer.?.send(0, packet);
                    try host.flush();
                    return;
                },
                .disconnect => return,
                .receive => {
                    defer event.packet.?.release();
                },
                else => {},
            }
        }
    }
}
```

Use the real file for reference:

14. Example: Creating a Reliable Packet

```zig
var packet = try zigenet.Packet.create(
    allocator,
    "payload",
    7,
    zigenet.constants.packet_flag_reliable,
);
defer packet.release();
packet.retain();
try peer.send(0, packet);
```

Important:
- retain before handing the packet into a queued send path when the local scope will also release it
- release received packets after processing them

16. Event Semantics

connect event
- emitted when the handshake reaches the connected state
- not every raw incoming connect command immediately becomes a connect event

disconnect event
- emitted when the peer disconnects or times out into disconnect behavior

receive event
- emitted when a payload is accepted, ordered if required, and dispatched
- event.packet must be released by the consumer

20. Guidance For LLMs

When answering questions:
- prefer facts from the current Zig source over assumptions from original ENet
- mention the new-packet flags when behavior differs between ordinary and Growtopia-oriented paths
- mention packet ownership rules when discussing Packet or Event.packet
- mention src/wire.zig when the user asks about exact bytes, sizes, headers, or protocol layout
- mention src/host.zig when the user asks about service loop, resend, ack, throttle, handshake, or dispatch

When making changes:
- keep the API Zig-native
- do not redesign public behavior unless necessary
- preserve Growtopia-specific support
- avoid describing behavior that is not implemented in code
- prefer adding tests when changing protocol behavior

When debugging:
- decide whether the problem is in transport, wire parsing, protocol state, or event dispatch
- check the new-packet flags first if the symptom only appears in Growtopia mode
- check checksum and compression hooks if received packets are rejected
